配信システム比較検討【詳細版】

チケット申込 × オンデマンド配信システムの3案を多角的に比較検討

作成日: 2026年1月24日

目次

これまでの構成（現行）

項目 サービス 備考
LP（ランディングページ） Wix 広報委員でも操作しやすいと想定して採用
チケット申込・管理 EventRegist Premium機能を利用
オンデマンド配信 Enavle EventRegistと連携

現行構成の課題

  • Wix: 広報委員が操作できるようにと導入したが、過去数年間、実質的に作業するのは事務局のみだった
  • EventRegist + Enavle: 案1で詳述する課題あり（代理購入不可、プレーヤー品質、高コスト等）
⚠️ 代理申し込み不可: 現行構成では各参加者が個別にチケットを購入する必要があり、法人が社員分を一括購入することができない。

今年度の方針

項目 方針 理由
LP Vercel（必要に応じてCMS導入） 事務局のみで運用するなら、より柔軟で低コストな構成が可能
チケット・配信 本資料で検討（案1〜3） 現行の課題を踏まえて最適な構成を選定

比較対象3案の概要

案 チケット申込 オンデマンド配信 連携方式
案1 EventRegist Enavle 統合（同一ベンダー）※制約あり
案2 任意のチケット管理サービス※ OneStream 分離（別サービス）
案3 microCMS/WordPress Cloudflare Stream/MUX 内製化

※案2のチケット管理サービスは、Peatix、EventRegist（Enavle連携なし）、Doorkeeper等から選択可能

⚠️ 案1の重要な制約: チケット購入は各参加者本人が個別にEventRegistで行う必要がある（代理での一括購入不可）。
理由: Enavleはシンガポールのサービスであり、本人以外の個人情報を海外サーバーに移転できないため（個人情報保護）。EventRegist登録後はEnavleに自動連携される。

ロール別フロー図

以下の4つのロールで各案の運用フローを視覚化:

  • 申込者: チケットを申し込む人（参加者を兼ねる場合あり、総務部等の代理申込も含む）
  • 参加者: 実際にイベントに参加・視聴する人
  • 事務局: 登録作業・事務作業・問い合わせ対応を行うJTFスタッフ
  • サービス: SaaS・サーバー等のシステム

現行 案1: EventRegist + Enavle（統合型）

サービス概要

  • EventRegist: 国内大手イベント管理プラットフォーム（5万団体以上が利用）
  • Enavle: シンガポール Actioport社開発のバーチャルイベントプラットフォーム

料金体系

項目金額
EventRegist基本機能無料
有料チケット決済手数料販売額の8%
Premium機能（enavle連携含む）550,000円〜/イベント
連携サービス利用費申込人数により変動

メリット

運用面

  • チケット販売〜視聴管理まで一元化
  • 視聴履歴の自動連携
  • 問い合わせ窓口の一本化

機能面

  • YouTube/Zoom等と連携可能
  • セッション毎のアンケート機能
  • スポンサー広告動画配信

デメリット

  • 高額な初期費用: Premium機能利用に55万円〜/イベント
  • 人数連動コスト: 大規模イベントでは費用が増大
  • Enavle独自のUIに依存（カスタマイズ制限）
  • ベンダーロックイン（他サービスへの移行が困難）

⚠️ 重要な制約：代理購入不可

  • チケット購入は各参加者本人が個別に行う必要がある
  • 総務部が社員分を一括購入することができない
  • 法人の代理購入ユースケースに対応できない
  • EventRegist登録後はEnavleに自動連携（Enavleへの再登録は不要）

理由: Enavleはシンガポールのサービスであり、本人以外の個人情報を海外サーバーに移転することは個人情報保護の観点から認められないため。

事務局コメント（数年間の運用実績に基づく評価）

JTF翻訳祭では過去数年間、案1（EventRegist + Enavle）を採用して運用してきた実績がある。以下は、その運用経験から得られた課題・所感である。

Enavleプレーヤーの問題点

  • 視聴プレーヤーの評判が悪い
  • UIや操作性に課題あり
  • 再生品質・安定性への懸念

ベンダー対応の問題点

  • フィードバックが改善に反映されない
  • 複数回要望しても目立った改善なし
  • 日本市場ニーズへの対応が不十分

→ 高額なシステム利用料に見合う価値が得られているか疑問。システムの見直しを検討する時期に来ている。

フォールバック 案2: チケット管理サービス + OneStream（分離型）

サービス概要

  • チケット管理: Peatix、EventRegist（Enavle連携なし）、Doorkeeper等から選択
  • OneStream: 株式会社ルートチーム運営の動画配信サイト構築プラットフォーム

料金体系

Peatix

初期費用・月額費用無料
有料チケット決済手数料4.9% + 99円/枚
振込手数料210円/イベント

OneStream

初期費用0円
スタータープラン9,800円〜/月
エンタープライズ要問合せ

メリット

  • 低い初期投資: Peatixは基本無料、OneStreamは月額9,800円〜
  • 導入ハードルが低い: 両サービスとも専門知識不要
  • サービスの独立性: 片方に問題があっても他方は継続可能

デメリット

  • 手動での参加者管理が必要: チケット購入者リストとOneStream会員の紐付け作業
  • 請求書払い非対応: 法人は別ルートでの対応が必要
  • 導線の分断: チケット購入→視聴ページへの誘導が複雑化
  • 問い合わせ窓口が分散
注意: Peatixは請求書払い非対応。法人からの一括購入には別途対応が必要（EventRegistは対応可）。

OneStream登録のタイミングによる運用の違い

タイミング登録方法運用
オンデマンド配信開始前 事務局が事前にID/パスワードを準備 参加者情報を収集後、事務局がOneStreamに一括登録
オンデマンド配信開始後 参加者が自身でサインアップ チケット自動返信メールに招待URLを記載
⚠️ セキュリティ上の懸念: 配信開始後の運用では、招待用URLがメールに記載されるため、このURLが外部に流出した場合、第三者がOneStreamに登録できてしまうリスクがある。

推奨・優先検討 案3: CMS + ストリーミングサービス（内製化）

開発手法: AIエージェント（Claude Code等）を活用した開発を前提とする。これにより、従来の外部委託や専任エンジニア確保に比べて、開発コストの大幅な削減と迅速な実装が可能。

サービス概要

  • フロントエンド: Next.js (App Router + TypeScript)
  • 認証・DB: Supabase
  • 動画配信: Cloudflare Stream
  • 決済: Stripe（3.6%）
  • メール: Resend

料金体系

microCMS

Hobby無料
Team4,900円/月
Business31,900円/月

Cloudflare Stream

動画保存$5/1,000分/月
動画配信$1/1,000分視聴

メリット

カスタマイズ性

  • JTFブランドに合わせた独自デザイン
  • 必要な機能だけを実装
  • API経由で任意のサービスと接続

コスト面

  • 決済手数料3.6%（Stripe）
  • 従量課金で初期投資を抑制
  • 大規模になるほど相対的に安価

データ管理

  • 参加者データを完全管理
  • 任意の指標でデータ分析可能

長期的視点

  • 翌年以降も再利用可能
  • 内部にノウハウ蓄積

デメリット（AIエージェント活用で軽減可能）

  • 開発リソースが必要: エンジニアまたは外部委託が必須 → AIエージェント活用により軽減
  • 構築期間: 数週間〜数ヶ月の開発期間 → AIエージェント活用により短縮可能
  • 保守・メンテナンス責任: 障害対応は自己責任

推奨プラグイン構成（WordPress）

  • FooEvents for WooCommerce ($49〜): 参加者別チケット発行、QRコード生成
  • Tickera: WooCommerce連携、20以上の決済ゲートウェイ対応
  • Event Tickets Plus: 各参加者の詳細情報収集、自動メール確認

総合評価と推奨アプローチ

案1を推奨するケース

  • • 予算に余裕がある場合
  • • スポンサー向け機能を重視
  • • 参加者が各自でEnavle登録を許容できる場合

※現行の課題を考慮すると非推奨

案2: フォールバック用

  • • 案3の開発が間に合わない場合
  • • 技術的課題が解決できない場合
  • • 確実に運営を実施したい場合

※案3が困難な場合の代替案

案3: 優先検討

  • • AIエージェント活用で低コスト開発
  • • 独自のブランディング・UXを実現
  • • 決済手数料を最小化（3.6%）
  • • 翌年以降も資産として再利用

推奨アプローチ（フォールバック戦略）

  1. まず案3（内製化）を検討・着手: AIエージェント（Claude Code等）を活用した開発で、従来より低コスト・短期間での実装を目指す
  2. 案3が困難な場合は案2にフォールバック: 開発が間に合わない場合は案2（チケット管理サービス+OneStream）で確実に運営
なぜ案3を先に検討するのか: 案2を先に採用すると、翌年度以降に案3への移行が検討されにくくなる傾向がある。内製化のメリット（コスト削減、カスタマイズ性、データ管理）を享受するためには、最初から案3を目標として取り組むことが重要。

案3の段階的実装

  • Phase 1: チケット販売は既存サービス（EventRegist Basic等）を利用、配信のみ内製（Cloudflare Stream）
  • Phase 2: 認証・会員管理を追加（Supabase Auth）
  • Phase 3: チケット販売も内製化（Stripe決済）
結論: いずれの場合も、案1（現行）の継続は非推奨。高コスト・低品質・改善されないという三重苦からの脱却が必要。
